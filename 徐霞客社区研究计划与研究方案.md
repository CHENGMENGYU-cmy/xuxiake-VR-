## 导言：四个核心概念

在展开研究计划之前，先厘清本报告中四个贯穿始终的核心概念——它们回答不同的问题，不可混用。

| 概念 | 回答的问题 | 核心产出 | 时间属性 |
|------|-----------|---------|---------|
| **任务（Task）** | 要研究什么？达成什么目标？ | 研究目标、关键成果、验收标准 | 目标导向，不定时间 |
| **方案（Solution）** | 怎么实现？用什么技术路线？ | 技术架构、接口设计、算法选型 | 方向性，不定时间 |
| **计划（Plan）** | 什么时候做？谁来做？ | 阶段划分、里程碑、资源分配 | 严格时间线 |
| **进度（Progress）** | 做到哪了？还差什么？ | 完成状态、遗留问题、下一步 | 实时快照 |

**四者关系**：任务定义"做什么"，方案描述"怎么做"，计划安排"何时做"，进度呈现"做到哪"。任务驱动方案，方案拆分计划，计划追踪进度，进度反馈修正任务——这是一个持续循环的管理闭环。

## 一、项目概述

### 1.1 项目定位

徐霞客系统是一个以 VR/AR 内容为核心的沉浸式旅行社交分享平台，让用户以第一人称视角记录和分享旅行体验。平台融合了 Instagram 的社交分享、YouTube 的视频内容、小红书的笔记功能、Strava 的运动轨迹等平台特色，打造差异化的 VR 旅行社区。

### 1.2 核心价值主张

- **沉浸式体验**：支持 VR360/VR180/空间视频，让用户足不出户体验世界各地风景
- **内容驱动社交**：围绕旅行内容建立社交关系，而非传统的聊天驱动
- **AR 增强现实**：通过 AR 眼镜数据接入，提供真实的增强现实旅行体验
- **社区化运营**：社群从群聊升级为内容社区，形成旅行爱好者聚集地

1. AR 眼镜数据格式调研（不同品牌/型号的数据输出差异）
2. 实时数据传输协议选型（WebSocket vs WebRTC vs 混合方案）
3. 数据预处理与压缩算法（视频编码、传感器数据降采样）
4. 边缘计算与云端协同架构（端侧预处理 vs 云端处理的分工）

**关键成果**：
- AR 眼镜数据接入 SDK
- 实时数据传输系统（延迟 < 200ms，丢包率 < 1%）
- 数据预处理管线

**验收标准**：
- 支持至少 3 种主流 AR 眼镜品牌
- 视频流传输延迟 < 200ms
- 传感器数据同步误差 < 50ms
- 丢包率 < 1%
- 弱网环境下可降级运行

#### 4.2.2 数据传输架构

```
┌──────────────┐     WebRTC (视频流)      ┌──────────────┐
│  AR 眼镜端   │ ◄──────────────────────► │  平台服务端   │
│              │     WebSocket (控制+传感)  │              │
│  - 视频采集  │ ◄──────────────────────► │  - 流媒体接收 │
│  - 传感器    │                           │  - 数据预处理 │
│  - GPS定位   │                           │  - 存储分发   │
│  - 本地缓存  │                           │  - 转码处理   │
└──────────────┘                           └──────────────┘
```

**传输策略**：
- **WebRTC** 负责视频流传输——低延迟、支持自适应码率、NAT穿透
- **WebSocket** 负责控制信令和传感器数据——可靠传输、低开销
- 弱网环境下自动切换为离线缓存模式（IndexedDB），恢复网络后批量同步

#### 4.2.3 数据处理管线

```
原始数据 → 预处理(降噪/去重) → 压缩(H.265编码/gzip) → 传输 → 云端解码 → 存储(OSS/CDN) → 分发
```

### 4.3 方案2：AR 内容渲染

#### 4.3.1 WebXR 集成架构

```typescript
class XRSessionManager {
  private session: XRSession | null = null;
  private renderer: THREE.WebGLRenderer;
  private scene: THREE.Scene;

  async startSession(mode: 'immersive-vr' | 'immersive-ar'): Promise<void> {
    if (!navigator.xr) {
      throw new Error('浏览器不支持 WebXR，切换到降级模式');
    }

    this.session = await navigator.xr.requestSession(mode, {
      requiredFeatures: ['local-floor', 'bounded-floor'],
      optionalFeatures: ['layers', 'hand-tracking'],
    });

    this.session.requestAnimationFrame(this.renderLoop.bind(this));
  }

  private renderLoop(time: number, frame: XRFrame): void {
    const pose = frame.getViewerPose(this.referenceSpace);
    if (pose) {
      // 将 AR 眼镜姿态数据同步到 3D 场景
      for (const view of pose.views) {
        this.renderer.render(this.scene, this.getCamera(view));
      }
    }
    this.session!.requestAnimationFrame(this.renderLoop.bind(this));
  }
}
```

#### 4.3.2 3D 渲染策略

| 内容类型 | 渲染引擎 | 说明 |
|---------|---------|------|
| VR360/VR180 视频 | Three.js (球体贴图) | 将等距柱状投影映射到球体内表面 |
| 3D 路线轨迹 | CesiumJS | 地球尺度下的 GPS 轨迹可视化 |
| AR 标记点 | Three.js (CSS3DRenderer) | 基于地理坐标叠加文字/图片/3D模型 |
| 沉浸式场景 | A-Frame (声明式) | 快速原型和简单场景 |

### 4.5 方案4：推荐算法优化

#### 4.5.1 多维度信号体系

```typescript
interface UserBehaviorSignal {
  userId: number;
  timestamp: number;

  // 显式信号（用户主动行为——高权重）
  explicit: {
    likes: number;        // 点赞数
    comments: number;     // 评论数
    shares: number;       // 分享数
    follows: number;      // 关注数
    saves: number;        // 收藏数

  // 隐式信号（用户被动行为——低权重，需去噪）
  implicit: {

  };

  // 社交信号（关系网络——中等权重）
  social: {
    mutualFriends: number;          // 共同好友数

    communityOverlap: number;       // 社群重叠度

  };
```

#### 4.5.2 混合推荐算法架构

```
用户请求推荐
    │
    ├── 召回层（多路召回，各召回 TopN）
    │   ├── 内容召回：标签/地点/媒体类型匹配
    │   ├── 协同召回：相似用户的行为聚合
    │   ├── 知识召回：用户明确偏好（兴趣标签）
    │   ├── 上下文召回：时间/地点/设备/天气
    │   └── 热门召回：时间衰减的热度兜底
    │
    ├── 排序层（精排——LR/GBDT/深度学习）
    │   ├── 特征工程：用户特征 + 内容特征 + 交叉特征
    │   └── 模型预测：CTR 预估 → 排序
    │
    └── 重排层（策略调整）
        ├── 多样性打散：避免同类型内容扎堆
        ├── 去重过滤：排除已浏览/已互动
        └── 负反馈学习：用户不感兴趣的内容降权
```

#### 4.5.3 实时推荐更新

- **流式处理**：用户行为通过 Redis Streams 进入实时计算管道
- **增量更新**：用户画像每 5 分钟更新，推荐结果随刷新请求动态调整
- **A/B 测试**：流量按 10%/10%/80% 分配至实验组A/实验组B/对照组

#### 4.6.2 后端优化策略

```typescript
// Redis 三级缓存策略
@Injectable()
class CacheService {
  // L1: 热点数据（TTL 60s）——首页动态、热门帖子
  async getFeedPosts(userId: number): Promise<Post[]> {
    const key = `feed:hot:${userId}`;
    let data = await this.redis.get(key);
    if (data) return JSON.parse(data);

    data = await this.postService.fetchFeedFromDB(userId);
    await this.redis.setex(key, 60, JSON.stringify(data));
    return data;
  }

  // L2: 温数据（TTL 300s）——帖子详情、用户主页
  // L3: 冷数据（TTL 3600s）——静态配置、历史归档
}
```

#### 4.6.3 数据库优化

- 针对高频查询建立复合索引（`idx_posts_user_created`, `idx_feed_composite`）
- 读写分离：主库写入，从库读取（Phase 2 引入）
- 大字段（GPX数据、富文本攻略正文）分离存储到 OSS

90% 的可能性意味着还有 10% 的失败空间。以下是可能导致开学前无法推出的真实场景，以及对应的降级策略。

**场景一：部署遇到不可逾越的障碍**

| 具体表现 | 云服务器申请不下来、域名备案耗时长、HTTPS 证书配置卡住 |
|---------|--------------------------------------------------|
| 降级方案 | 前端部署到 Vercel（免费），后端用 Ngrok 隧道代理本机服务，先用临时域名演示 |
| 底线交付 | 至少能在浏览器中打开一个可交互的页面，哪怕后端在笔记本上跑 |
| 影响评估 | 用户不能自由访问，但可以做现场演示和录屏展示 |

**场景二：测试阶段发现严重 Bug**

| 具体表现 | 核心流程（注册/发帖/聊天）有阻断性 bug，无法在短期内修复 |
|---------|--------------------------------------------------------|
| 降级方案 | 关闭有问题的子功能入口（如社群挑战、语音消息），保留主链路干净可用 |
| 底线交付 | 注册 + 浏览内容 + 发帖 + 点赞评论 + 基础聊天 五条主线必须跑通 |
| 影响评估 | 功能缩水但不影响核心体验闭环，后续通过发版逐步恢复 |

**场景三：人手不足或时间被挤压**

| 具体表现 | 成员请假、暑假其他安排冲突、实际可用开发时间远少于6周 |
|---------|----------------------------------------------------|
| 降级方案 | 砍掉所有"锦上添花"的工作（移动适配、推荐调优、种子数据），只保留部署+测试两件事 |
| 底线交付 | 后端 + 前端部署到公网可访问，哪怕功能不完美 |
| 影响评估 | 产品粗糙但可用，开学后利用课余时间持续迭代 |

**场景四：开学前完全无法交付公网版本**

| 具体表现 | 以上所有降级方案均失败，没有任何可公网访问的版本 |
|---------|----------------------------------------------|
| 兜底方案 | 本地 localhost 版本做完整演示（录屏 + 现场操作），整理项目文档，输出一份"当前可演示功能清单" |

| 影响评估 | 虽然不能上线，但项目成果可以完整展示，不会"什么都拿不出来" |

```typescript
describe('AR 眼镜数据接入', () => {

    expect(devices.length).toBeGreaterThan(0);

    await networkSimulator.setCondition('slow-3g');

    expect(result.mode).toBe('offline-cache');
  });
});
```

## 七、风险分析与应对

### 7.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| AR 眼镜兼容性差异大 | 高 | 高 | 设计统一抽象层；优先适配主流品牌；建立兼容性测试矩阵 |
| WebXR 浏览器支持不足 | 中 | 中 | WebXR Polyfill 兜底；提供非 XR 降级模式；持续跟踪标准演进 |
| 高并发下 WebSocket 瓶颈 | 中 | 高 | 连接池 + 消息压缩 + 优雅降级至轮询 |
| 推荐算法冷启动 | 高 | 中 | 基于内容+热门兜底；新用户兴趣快速问卷；渐进式学习 |

### 7.2 项目风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| AR 接入复杂度超预期 | 中 | 高 | 预留 20% 缓冲时间；分阶段交付；备选技术方案 |
| 设备资源不足 | 中 | 中 | 提前采购；设备共享；模拟数据先行开发 |
| 用户对 AR 功能接受度低 | 低 | 高 | 渐进式引导；非 AR 替代体验；用户调研驱动迭代 |

### 7.3 市场风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 竞品推出类似功能 | 中 | 中 | 深耕旅行垂直领域；社区壁垒；持续创新 |
| AR 硬件普及度不足 | 高 | 中 | 同时支持手机端内容创作；降低 AR 内容生产门槛 |
